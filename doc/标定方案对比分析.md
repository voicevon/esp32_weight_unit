# 标定方案对比分析 (Calibration Strategy Analysis)

本文档对比分析了两种称重标定与计算方案：**方案一：本地计算 (Local Calculation)** 与 **方案二：远程计算 (Remote Calculation)**。

## 1. 方案概述

### 方案一：本地计算 (Local Calculation) - **推荐**
- **原理**: ESP32 读取传感器原始值(AD)，在本地进行去皮、系数转换，直接输出**重量值(克)**。
- **数据流**: Sensor -> AD -> [ESP32: (AD-Offset)/Scale] -> Weight(g) -> RS485 -> PC
- **特点**: ESP32 下位机独立性强，直接提供最终结果。

### 方案二：远程计算 (Remote Calculation)
- **原理**: ESP32 仅作为“采集卡”，读取传感器原始值(AD)并通过 RS485 透传给上位机。上位机维护所有标定参数并计算重量。
- **数据流**: Sensor -> AD -> [ESP32: Raw AD] -> RS485 -> [PC: (AD-Offset)/Scale] -> Weight(g)
- **特点**: 逻辑集中在上位机，下位机功能极简。

## 2. 优劣势对比 (Pros & Cons)

| 维度 | 方案一：本地计算 (分布式) | 方案二：远程计算 (集中式) |
| :--- | :--- | :--- |
| **实时性 (Real-time)** | **高**。OLED 显示屏可立即显示重量，无延迟。 | **低**。OLED 需要等待 PC 计算后回传数据才能显示，会有明显滞后。 |
| **独立性 (Independence)** | **强**。单个模块可独立工作、测试、显示，不依赖上位机。 | **弱**。离开上位机，模块无法显示重量，仅能显示乱码或AD值。 |
| **系统负载 (Load)** | **均衡**。20+个节点的计算压力分散在各个 ESP32 上。 | **集中**。上位机需轮询读取所有 AD 值并计算，通信带宽占用较高。 |
| **标定流程 (Calibration)** | **分散**。需对每个节点发送标定指令，参数存在 ESP32 本地。 | **集中**。参数统一存在数据库/文件中，更换模块无需重新标定(只需更新参数)。 |
| **维护性 (Maintenance)** |更换模块需重新标定或写入旧参数。 | 更换模块方便，但依赖上位机软件稳定性。 |

## 3. 标定流程对比 (Calibration Process)

### 方案一流程 (本地计算)
1.  **零点标定 (Zero)**:
    - 确保秤台空载。
    - 上位机发送指令 `TARE`。
    - ESP32 读取当前 AD 值作为 `Offset` 并保存到 Flash。
    - 返回 `OK:TARED`。
2.  **重量标定 (Weight)**:
    - 放置已知重量砝码 (例如 1000g)。
    - 上位机发送指令 `SET_SCALE:系数` 或 `CAL:1000`。
    - ESP32 计算系数 `Scale = (CurrentAD - Offset) / 1000` 并保存到 Flash。
    - 返回 `OK:CALIBRATED`。

### 方案二流程 (远程计算)
1.  **零点标定 (Zero)**:
    - 确保秤台空载。
    - 上位机读取当前 AD 值 (Cmd: `GET_AD`)。
    - 上位机将该 AD 值记录为 `Offset`，存入数据库。
2.  **重量标定 (Weight)**:
    - 放置已知重量砝码 (1000g)。
    - 上位机读取当前 AD 值。
    - 上位机计算 `Scale = (CurrentAD - Offset) / 1000`，存入数据库。

## 4. 结论与建议
由于本项目包含 **OLED 本地显示** 需求，**强烈推荐使用方案一（本地计算）**。
- 若采用方案二，OLED 显示内容必须由 PC 计算后回传，在 RS485 总线上 20+ 个节点轮询的环境下，显示刷新率将极低且不仅不实时，甚至可能完全无法满足视觉需求。
- 方案一虽然增加了下位机代码复杂度（需保存参数），但极大降低了通信与系统耦合度，符合工业现场“边缘计算”的趋势。
